# 场景匹配索引

> 经验不是死的，是活的。遇到新场景时，要能快速找到相关的经验。

---

## 索引原则

### 核心概念：场景 = 行动 + 情境

```
场景不是"发邮件"
场景是"给外部合作方发邮件"

场景不是"写报告"
场景是"写月度项目进展报告，给领导看"

场景不是"开会"
场景是"和外部合作方开线上会议，需要做记录"
```

**为什么要这样分**：
- 同样"发邮件"，给内部同事和给外部客户的做法可能完全不同
- 同样"写报告"，给领导看和给客户看格式完全不同
- 把场景分清楚，才能准确匹配经验

## 匹配方法

### 1. 精确匹配

当新场景**完全等于**已有场景时：
```
新场景：给外部合作方发邮件
匹配到：📄 给外部合作方发邮件
→ 直接应用所有相关经验
```

### 2. 父级匹配

当新场景**没有精确匹配**但有父级场景时：
```
新场景：给外国合作方发邮件（没有这个场景）
匹配到：📁 邮件（父级）
→ 应用父级通用经验 + 特别标注"需要确认细节"
```

### 3. 标签匹配

当**多个标签**匹配到不同场景时：
```
新场景：给新客户发邮件，需要正式且简洁
匹配到：
  - 📄 给外部合作方发邮件（标签：#外部沟通 #正式）
  - 📄 给领导发邮件（标签：#简洁）
→ 合并两个场景的经验，冲突时优先精确匹配
```

### 4. 情境匹配

当遇到**新场景但有类似情境**时：
```
新场景：给投资人发邮件（没有这个场景）
匹配到：
  - 📄 给外部合作方发邮件（外部、正式）
  - 📄 给领导发邮件（需要CC、简洁）
→ 应用类似情境的经验，但主动询问确认
```

## 匹配算法（伪代码）

```
function match_scene(new_scenario):
    # 1. 提取关键词
    keywords = extract_keywords(new_scenario)
    
    # 2. 精确匹配
    exact_match = search_exact(keywords)
    if exact_match:
        return exact_match
    
    # 3. 父级匹配
    parent_match = search_parent(keywords)
    if parent_match:
        return parent_match + flag("需要确认细节")
    
    # 4. 标签匹配
    tag_matches = search_by_tags(keywords)
    if tag_matches:
        return merge_and_prioritize(tag_matches) + flag("跨场景经验")
    
    # 5. 情境匹配
    similar = search_similar_context(keywords)
    if similar:
        return similar + flag("类似场景，请确认")
    
    # 6. 完全没匹配
    return None + flag("新场景，需要建立经验")
```

## 场景描述模板

建立新场景时，用这个模板：

```markdown
## 场景：[场景名称]

**场景要素**：
- 谁做的：主人 / 我 / 主人和我
- 做什么：具体动作
- 对象是谁：内部同事 / 外部合作 / 领导 / 客户
- 什么形式：文字 / 口头 / 正式 / 非正式
- 预期目标：沟通 / 汇报 / 决策 / 记录

**触发条件**：什么情况下会进入这个场景

**关联动作**：
- 这个场景之前通常会做什么
- 这个场景之后通常会做什么

**相关场景**：
- [场景A] — [为什么相关]
- [场景B] — [为什么相关]
```

## 示例

### 示例1：建立新场景

```markdown
## 场景：给客户发项目报价邮件

**场景要素**：
- 谁做的：主人（我来起草）
- 做什么：发邮件给客户，附报价
- 对象是谁：外部客户
- 什么形式：邮件，正式
- 预期目标：让客户接受报价，推进合作

**触发条件**：
- 主人说"帮我给XX发个报价"
- 项目谈得差不多了，要正式报价

**关联动作**：
- 之前：可能需要先和主人确认报价范围
- 之后：可能要等客户反馈，可能需要修改

**相关场景**：
- 📄 给外部合作方发邮件 — 通用外部沟通规范
- 📄 给领导发邮件 — 需要CC主人
- 📄 写项目报价 — 具体怎么写报价

**新经验待补充**：
- 报价邮件的格式偏好？
- 需要附件还是正文？
- 语气要正式到什么程度？
```

### 示例2：场景匹配过程

```
新场景：老板让我给一个新客户发邮件介绍一下我们的服务

# 匹配过程

1. 精确匹配？
   → 没有"给新客户发介绍邮件"这个场景
   → 继续

2. 父级匹配？
   → 有"📁 邮件 > 📄 给外部合作方发邮件"
   → 匹配到：
     - 需要CC老板
     - 语气正式
     - 发送前需要老板确认
   → 但这是"合作方"不是"客户"，可能有差异

3. 标签匹配？
   → #外部 #正式 #介绍
   → 没有特别匹配

4. 情境匹配？
   → "给新客户介绍服务" 类似 "给合作方发邮件"
   → 但客户可能需要更多背景介绍

# 结论
→ 应用"给外部合作方发邮件"的通用规则
→ 额外主动询问：
  - "这是新客户还是之前联系过的？"
  - "需要附上我们的服务介绍资料吗？"
  - "这次邮件你希望突出什么？"
```

## 维护指南

### 什么时候拆分子场景

```
当一个场景内的经验开始互相矛盾时

例如"给外部合作方发邮件"：
- 经验1：合作方A要求详细，主人喜欢详细
- 经验2：合作方B要求简洁，主人喜欢简洁
→ 矛盾！
→ 拆分为：
  - 📄 给外部合作方A发邮件
  - 📄 给外部合作方B发邮件
```

### 什么时候合并场景

```
当多个场景的经验完全一致时

例如：
- 📄 给领导发邮件
- 📄 给老板发邮件
- 给大领导发邮件
→ 如果这三个场景的经验完全一致
→ 合并为一个场景"📁 给上级发邮件"
```

### 什么时候删除场景

```
当一个场景完全没有相关经验，且6个月以上没有被触发时
→ 标记为"休眠场景"
→ 如果之后被触发，重新激活并补充经验
```

## 快速参考

### 匹配优先级

```
1. 精确匹配 > 父级匹配 > 标签匹配 > 情境匹配 > 新建

2. 精确匹配有多条？
   → 选择场景描述最详细的那个

3. 多个标签匹配有冲突？
   → 优先"对象"（给谁做）> "形式"（怎么做）> "目的"（为什么做）
```

### 遇到新场景怎么办

```
1. 先想：有没有类似的场景？
2. 如果有：先应用类似场景的经验
3. 然后：主动问主人，这个情况有什么特别的吗？
4. 最后：把新经验记下来
```
